## Project Overview

memex is a self-hosted, generic RAG (Retrieval-Augmented Generation) system built
around PostgreSQL + pgvector. It exposes a REST API for document ingestion and
semantic search, and serves a Vue frontend for document management.

It is intentionally generic — not tied to any domain. The owner uses it as a
shared RAG backend for multiple projects (food safety agents, MCP servers, etc).

---

## Goals

- Simple to deploy: single docker-compose up and everything works
- Generic: any document type, any domain
- Clean API: well-documented Swagger (swaggo/swag), ready for MCP integration
- High code quality: DRY, KISS, idiomatic Go, documented functions
- Production-ready logging: slog with console and JSON modes
- Resilient ingestion: retries with exponential backoff, graceful failure handling

## Technology Decisions

### Backend
- Language: Go (same stack as owner's other MCP projects)
- HTTP router: net/http standard library + chi router (lightweight, idiomatic)
- Swagger: swaggo/swag — run `swag init -g cmd/main.go` from server/ to regenerate docs
- Logger: log/slog — single instance, passed via context or dependency injection
  - LOG_FORMAT=console → human-readable (default for dev)
  - LOG_FORMAT=json → structured JSON (for production/k8s)
- DB driver: pgx/v5 (not database/sql) — native pgvector support, better performance

### Frontend
- Framework: Vue 3 (Composition API)
- Build tool: Vite
- Embedded in Go binary via embed.FS — zero separate static file server needed
- Served by Go at GET / and GET /app/*

### Database
- PostgreSQL with pgvector extension
- Image: pgvector/pgvector:pg16 (pgvector pre-installed)
- The db/ package runs migrations on startup (no external migration tool)
- Schema:
  - documents table: id, filename, format, status, error, chunk_count, created_at, updated_at
  - chunks table: id, document_id, content, embedding vector(N), chunk_index, metadata jsonb

### Embeddings
- Provider: Ollama (owner already has it deployed)
- Called via HTTP from the embedder/ package
- Model is configurable via OLLAMA_MODEL env var (default: nomic-embed-text)
- IMPORTANT: the same model must be used for ingestion and search — enforced by config

### Vector Search
- pgvector cosine similarity search
- Returns top K chunks (configurable, default 5) with similarity score and source metadata
- No reranking in v1

## Supported Document Formats

Implemented in server/internal/parser/, one file per format.
All parsers implement the Parser interface:

```go
type Parser interface {
    Parse(r io.Reader) (string, error)
    Extensions() []string
}
```

Supported in v1:
- PDF (.pdf) — pdfcpu or go-pdfium
- Plain text (.txt)
- Markdown (.md, .markdown)
- DOCX (.docx) — unioffice or gooxml
- ODT (.odt) — ZIP+XML, no external lib needed
- HTML (.html, .htm) — golang.org/x/net/html
- CSV (.csv) — encoding/csv (stdlib)
- JSON (.json) — encoding/json (stdlib)
- YAML (.yaml, .yml) — gopkg.in/yaml.v3
- TOML (.toml) — BurntSushi/toml
- XML (.xml) — encoding/xml (stdlib)
- RTF (.rtf) — basic parser, strip control words
- EML (.eml) — net/mail (stdlib)

Deferred to v1.5:
- PPTX, XLSX — feasible but adds complexity, not in v1

## Ingestion Pipeline

```
Upload → Parser (extract text) → Chunker (split) → Embedder (Ollama) → DB (pgvector)
```

The worker is a goroutine pool (configurable size, default 3) reading from an
in-memory job channel. The API enqueues jobs and returns immediately.

### Retry Logic
- Max attempts: 3 (configurable via WORKER_MAX_RETRIES)
- Backoff: exponential — 2s, 4s, 8s
- On exhaustion: document status → "failed", error message stored in DB, worker moves on
- Failed documents are logged at ERROR level with full context (document_id, filename, attempt, error)
- The job queue never blocks: if full, upload returns 429 Too Many Requests

### Document Status Flow
```
pending → processing → completed
                    ↘ failed
```

## API Design

Base path: /api/v1
Swagger UI: /swagger/index.html

### Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | /api/v1/documents | Upload and enqueue a document |
| GET | /api/v1/documents | List all documents (filterable by status) |
| GET | /api/v1/documents/{id} | Get document detail and status |
| DELETE | /api/v1/documents/{id} | Delete document and all its chunks |
| POST | /api/v1/search | Semantic search over indexed chunks |
| GET | /api/v1/health | Health check (DB + Ollama connectivity) |

### Search Request/Response

```json
POST /api/v1/search
{
  "query": "what are the maximum aflatoxin levels in cereals",
  "limit": 5
}

Response:
{
  "results": [
    {
      "chunk_id": "uuid",
      "document_id": "uuid",
      "filename": "codex_contaminants.pdf",
      "chunk_index": 12,
      "content": "...",
      "score": 0.91
    }
  ]
}
```

### Security (planned, not in v1)
- API key authentication via X-API-Key header
- Middleware hook already in place in router.go, just needs enabling

## Configuration (Environment Variables)

| Variable | Default | Description |
|----------|---------|-------------|
| PORT | 8080 | HTTP server port |
| LOG_FORMAT | console | console or json |
| LOG_LEVEL | info | debug, info, warn, error |
| DATABASE_URL | (required) | PostgreSQL DSN with pgvector |
| OLLAMA_URL | http://localhost:11434 | Ollama base URL |
| OLLAMA_MODEL | nomic-embed-text | Embedding model name |
| OLLAMA_EMBEDDING_DIM | 768 | Vector dimensions (must match model) |
| WORKER_POOL_SIZE | 3 | Concurrent ingestion workers |
| WORKER_MAX_RETRIES | 3 | Max attempts per document |
| CHUNK_SIZE | 512 | Target chunk size in tokens |
| CHUNK_OVERLAP | 64 | Overlap between chunks in tokens |
| SEARCH_DEFAULT_LIMIT | 5 | Default number of results returned |
| MAX_UPLOAD_SIZE_MB | 50 | Maximum upload file size |

## Docker & CI/CD

### Dockerfile
Multi-stage build:
1. Stage 1 (node): build Vue frontend → dist/
2. Stage 2 (go-builder): embed dist/ + build Go binary
3. Stage 3 (final): distroless/scratch image, only the binary

### docker-compose.yml
Services:
- postgres: pgvector/pgvector:pg16, with volume for data persistence
- memex: the application, depends_on postgres with healthcheck

### GitHub Actions

**.github/workflows/release.yml** — triggered on git tag (v*)
- Builds binary for: linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64
- Uploads all binaries to GitHub Release as assets
- Generates checksums file (sha256)

**.github/workflows/docker.yml** — triggered on git tag (v*) and push to main
- Builds multi-arch Docker image (linux/amd64, linux/arm64)
- Pushes to ghcr.io/{owner}/memex:{tag} and ghcr.io/{owner}/memex:latest

## Code Quality Rules

These rules are non-negotiable and must be followed in every file:

1. Every exported function and type must have a GoDoc comment
2. Every non-trivial function must have a header comment explaining what it does,
   its parameters, and what it returns or mutates
3. No business logic in handlers — handlers only parse input, call service/domain
   functions, and write responses
4. Errors are always wrapped with context: fmt.Errorf("doing X: %w", err)
5. No global variables except the logger (and even that is passed via context when possible)
6. Config is read once at startup and passed as dependency, never read from env mid-execution
7. All magic numbers are named constants
8. DRY: if something is written twice, it becomes a function
9. KISS: the simplest solution that works correctly is always preferred
