## Project Overview

memex is a self-hosted, generic RAG (Retrieval-Augmented Generation) system built
around PostgreSQL + pgvector. It exposes a REST API for document ingestion and
semantic search, and serves a Vue frontend for document management.

It is intentionally generic — not tied to any domain. The owner uses it as a
shared RAG backend for multiple projects (food safety agents, MCP servers, etc).

---

## Goals

- Simple to deploy: single docker-compose up and everything works
- Generic: any document type, any domain
- Multi-tenant: namespace isolation, API key auth, per-key namespace access
- Clean API: well-documented Swagger (swaggo/swag), ready for MCP integration
- High code quality: DRY, KISS, idiomatic Go, documented functions
- Production-ready logging: slog with console and JSON modes
- Resilient ingestion: retries with exponential backoff, graceful failure handling

## Technology Decisions

### Backend
- Language: Go (same stack as owner's other MCP projects)
- HTTP router: net/http standard library + chi router (lightweight, idiomatic)
- Config: YAML file passed via `-config` flag; `${ENV_VAR}` expansion supported
- Swagger: swaggo/swag — run `swag init -g cmd/main.go` from server/ to regenerate docs
- Logger: log/slog — single instance, passed via context or dependency injection
  - `log.format=console` → human-readable (default for dev)
  - `log.format=json` → structured JSON (for production/k8s)
- DB driver: pgx/v5 (not database/sql) — native pgvector support, better performance

### Frontend
- Framework: Vue 3 (Composition API)
- Build tool: Vite
- Embedded in Go binary via embed.FS — zero separate static file server needed
- Served by Go at `GET /` and `GET /*`

### Database
- PostgreSQL with pgvector extension
- Image: `pgvector/pgvector:pg16` (pgvector pre-installed)
- The db/ package runs inline idempotent migrations on startup (no external migration tool)
- Schema:
  - `documents` table: id, namespace, filename, format, status, error, chunk_count, file_hash, file_path, instance_id, created_at, updated_at
  - `chunks` table: id, document_id, namespace, content, embedding vector(N), chunk_index, metadata jsonb

### Embeddings
- Provider: any OpenAI-compatible API (owner uses Ollama)
- Called via HTTP from the embedder/ package
- Model and base_url are configurable in the YAML config
- IMPORTANT: the same model must be used for ingestion and search — enforced by config

### Vector Search
- pgvector cosine similarity search with IVFFlat index
- Namespace-scoped: `WHERE namespace = $1` (or no filter if namespace is "")
- Returns top K chunks (configurable, default 5) with similarity score and source metadata
- No reranking in v1

## Namespaces

Namespaces are deeply integrated. Every document and chunk carries a `namespace TEXT`
field. All API operations (upload, list, get, delete, search) are scoped to the namespace
provided in the `X-Memex-Namespace` request header.

### Config
```yaml
namespaces:
  - name: food-safety
  - name: legal

auth:
  api_keys:
    - key: "secret-key-1"
      namespaces: ["food-safety"]
    - key: "admin-key"
      namespaces: ["*"]   # access to all declared namespaces
```

When `namespaces` is empty → open mode, any namespace string is accepted.
When `auth.api_keys` is empty → auth is disabled, all requests are allowed through.

### Behaviour
- Requests for undeclared namespaces → `400 Bad Request`
- Valid key but wrong namespace → `403 Forbidden`
- Missing key when auth enabled → `401 Unauthorized`
- `db.Search` with `namespace=""` searches across all namespaces (used internally, not exposed)

---

## Authentication

API key authentication via two request headers:
- `X-Memex-Api-Key` — the API key
- `X-Memex-Namespace` — the target namespace

Two middleware variants:
- `mw.Auth` — full auth (key + namespace); used on all data endpoints
- `mw.AuthKeyOnly` — key only, no namespace required; used on `GET /api/v1/info`

Auth is opt-in: when `auth.api_keys` is empty in the config, both middleware variants
pass all requests through without checking headers.

## MCP Server

The MCP server lives at `mcp/` as a **separate Go module** (`module memex-mcp`).
It communicates with the Memex REST API over HTTP — it does not import server internals.
Transport: Streamable HTTP (modern MCP over HTTP + SSE) or stdio.

### Tools (6 total)
| Tool | Required | Optional |
|------|----------|----------|
| `list_documents` | — | `namespace`, `status` |
| `get_document` | `id` | `namespace` |
| `upload_document` | `path` | `namespace` |
| `delete_document` | `id` | `namespace` |
| `search` | `query` | `namespace`, `limit` |
| `health` | — | — |

Tool names are prefixed with a sanitized version of the configured server name.

### Namespace resolution (precedence)
1. Tool argument `namespace`
2. Forwarded `X-Memex-Namespace` header from the agent's request (if `forward_namespace: true`)
3. `memex.default_namespace` from config

### Auth forwarding
- `forward_api_key: true` → forwards the agent's `X-Memex-Api-Key` header to Memex
- `namespace_keys` map → per-namespace static API keys; `"*"` is the catch-all

### Security middleware
- JWT validation with JWKS endpoint + CEL expression evaluation
- Tool policy enforcement via CEL rules (per-tool access control)
- Both middlewares are optional (no-op when not configured)

## Ingestion Pipeline

```
Upload → Parser (extract text) → Chunker (split) → Embedder (API) → DB (pgvector)
```

The worker is a goroutine pool (configurable size, default 3) reading from an
in-memory job channel. The API enqueues jobs and returns 202 immediately.

### Retry Logic
- Max attempts: configurable via `worker.max_retries` (default 3)
- Backoff: exponential — 2s, 4s, 8s
- On exhaustion: document status → "failed", error message stored in DB
- Failed documents are logged at ERROR level with full context

### Document Status Flow
```
pending → processing → completed
                    ↘ failed
```

### Crash Recovery
On startup, the worker scans for documents in `processing` state belonging to this
instance (`instance_id`) and re-enqueues them. Documents stuck from a previous crash
are retried automatically.

### Deduplication
Files are hashed on upload. If a document with the same `(namespace, file_hash)` already
exists and is not failed, the upload returns the existing document instead of re-ingesting.

Multi-stage build:

## Code Quality Rules

These rules are non-negotiable and must be followed in every file:

1. Every exported function and type must have a GoDoc comment
2. Every non-trivial function must have a header comment explaining what it does,
   its parameters, and what it returns or mutates
3. No business logic in handlers — handlers only parse input, call service/domain
   functions, and write responses
4. Errors are always wrapped with context: `fmt.Errorf("doing X: %w", err)`
5. No global variables except the logger (and even that is passed via context when possible)
6. Config is read once at startup and passed as dependency, never read from env mid-execution
7. All magic numbers are named constants
8. DRY: if something is written twice, it becomes a function
9. KISS: the simplest solution that works correctly is always preferred
